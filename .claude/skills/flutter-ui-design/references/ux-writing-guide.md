# UX Writing Guide

アプリ内のテキストを書く際の原則とパターン。

---

## 基本原則

### 1. 明確さ（Clarity）
- 専門用語を避ける
- 一つの文で一つのことを伝える
- 曖昧な表現を避ける

### 2. 簡潔さ（Conciseness）
- 不要な言葉を削る
- 最も短い表現を選ぶ
- ただし必要な情報は省略しない

### 3. 有用性（Usefulness）
- ユーザーが次に何をすべきかを伝える
- 問題解決の手段を提示する
- 状況に応じた情報を提供する

### 4. ブランドの声（Brand Voice）
- 一貫したトーンを維持
- アプリの性格に合った言葉選び
- 親しみやすさと信頼感のバランス

---

## ボタン・CTA

### 原則
- 動詞で始める（アクションを明確に）
- 結果がわかる言葉を選ぶ
- 2〜4語に収める

### 例
| ❌ 避ける | ✅ 推奨 | 理由 |
|----------|--------|------|
| はい | 保存する | 何をするか明確 |
| OK | 送信 | アクションが具体的 |
| 次へ | プロフィールを作成 | ゴールが見える |
| Submit | 予約を確定 | 日本語で、結果が明確 |
| Click here | 詳細を見る | アクションが具体的 |

### CTAの階層
```
Primary:   「はじめる」「登録する」「購入する」
Secondary: 「詳細を見る」「キャンセル」
Tertiary:  「あとで」「スキップ」
```

## エラーメッセージ

### 原則
1. 何が起きたかを説明
2. なぜ起きたかを説明（可能なら）
3. どう解決するかを提示

### テンプレート
```
[何が起きたか]。[解決方法]。
```

### 例
| ❌ 避ける | ✅ 推奨 |
|----------|--------|
| エラーが発生しました | 接続できませんでした。インターネット接続を確認してください |
| 入力が無効です | パスワードは8文字以上で入力してください |
| Error 404 | ページが見つかりませんでした。URLを確認するか、トップページに戻ってください |
| 失敗しました | 保存できませんでした。もう一度お試しください |
| 不正な値です | メールアドレスの形式が正しくありません |

### エラーのトーン
- 責めない（「あなたの入力が間違っています」→「形式が異なります」）
- 技術用語を避ける（「500 Internal Server Error」→「問題が発生しました」）
- 解決策を提示する

## 空の状態（Empty State）

### 構成要素
1. 状況の説明
2. 次のアクションへの誘導
3. （オプション）イラスト

### 例
```
📭 まだメッセージはありません

マッチした相手にメッセージを送ってみましょう。
会話が始まるとここに表示されます。

[マッチを探す]
```

```
🔍 検索結果がありません

「{検索キーワード}」に一致する結果は見つかりませんでした。
別のキーワードで試してみてください。

[検索条件をクリア]
```

## 確認ダイアログ

### 原則
- タイトルは行動を示す
- 本文は結果を説明
- ボタンは具体的なアクション

### 例
```
❌ 避ける
-----------
タイトル: 確認
本文: 本当によろしいですか？
ボタン: [はい] [いいえ]

✅ 推奨
-----------
タイトル: アカウントを削除しますか？
本文: この操作は取り消せません。
      全てのデータが完全に削除されます。
ボタン: [削除する] [キャンセル]
```

### 破壊的アクションの場合
- 赤いボタンで危険性を示す
- 結果を具体的に説明
- 確認入力を求める（「削除」と入力）

## ローディング・進捗

### 原則
- 何が起きているかを伝える
- 待ち時間の目安を示す（可能なら）
- 退屈させない

### 例
| シーン | テキスト |
|--------|---------|
| 一般的 | 読み込み中... |
| データ取得 | 最新の情報を取得しています |
| 送信中 | メッセージを送信しています |
| 長い処理 | あと少しで完了します（約30秒） |
| 初回起動 | あなた専用の設定を準備しています |

---

## フォームラベル・プレースホルダー

### 原則
- ラベルは必須（プレースホルダーだけに頼らない）
- プレースホルダーは入力例を示す
- ヘルプテキストで補足

### 例
```
ラベル:       メールアドレス
プレースホルダー: example@email.com
ヘルプテキスト:  ログインに使用します

ラベル:       パスワード
プレースホルダー: ••••••••
ヘルプテキスト:  8文字以上、英数字を含む
```

## 通知・アラート

### 種類別のトーン
| 種類 | トーン | 例 |
|------|-------|-----|
| 成功 | ポジティブ、簡潔 | 「保存しました」「送信完了」 |
| 情報 | 中立、説明的 | 「新機能が追加されました」 |
| 警告 | 注意を促す | 「保存されていない変更があります」 |
| エラー | 解決志向 | 「接続できませんでした。再試行してください」 |

## オンボーディング

### 原則
- ベネフィットを伝える（機能説明ではない）
- 1画面1メッセージ
- スキップ可能に

### 例
```
❌ 機能説明
「スワイプ機能を使って、左右にカードを動かすことができます」

✅ ベネフィット
「直感的な操作で、あなたにぴったりの相手を見つけましょう」
```

## 日本語特有の注意点

### 敬語レベル
- です/ます調を基本に
- 過度な敬語は避ける
- 一貫性を保つ

### 長さの調整
- 日本語は英語より短くなることが多い
- UIの幅を考慮
- 改行位置に注意

### 表現の選択
| 英語風 | 自然な日本語 |
|--------|-------------|
| ログインする | ログイン |
| サブミットする | 送信 |
| キャンセルしますか？ | キャンセルしてもよいですか？ |

## チェックリスト

- [ ] 動詞で始まるボタンラベル
- [ ] エラーには解決策がある
- [ ] 専門用語を使っていない
- [ ] 敬語レベルが統一されている
- [ ] 空の状態に次のアクションがある
- [ ] 確認ダイアログは具体的
- [ ] プレースホルダーだけに頼っていない
